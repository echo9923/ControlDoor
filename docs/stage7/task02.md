# 阶段 7 / 任务 02：回调接收与事件队列

## 目标

在阶段 3.7 报警布防基础上实现 ACS 回调接收和队列投递。SDK 回调线程必须尽快返回，只负责复制必要数据、识别来源设备并把原始事件投递到后台队列。

## 输入

SDK 回调提供的核心信息：

| 字段 | 说明 |
| --- | --- |
| `lCommand` | 命令号，ACS 事件为 `COMM_ALARM_ACS`。 |
| `NET_DVR_ALARMER` | 来源设备 IP、UserID、序列号等。 |
| `pAlarmInfo` | 指向 `NET_DVR_ACS_ALARM_INFO` 的指针。 |
| `dwBufLen` | 报警结构长度。 |
| `pUser` | 回调上下文。 |

## 回调线程规则

| 规则 | 说明 |
| --- | --- |
| 只复制 | 复制报警结构原始字节、图片字节和来源信息。 |
| 不写库 | 不访问 `attendance_gate_v2` 或 `face_event_checkpoint`。 |
| 不写文件 | 不保存抓拍图片。 |
| 不阻塞 | 队列满时按降级策略处理，不能长时间等待。 |
| 不抛异常 | 捕获所有异常并记录，避免影响 SDK 回调链路。 |

## 设备来源识别

优先级：

| 优先级 | 来源 | 说明 |
| --- | --- | --- |
| 1 | `NET_DVR_ALARMER.sDeviceIP` | 首选设备上报 IP。 |
| 2 | `lUserID` 反查 | IP 为空时用阶段 4 运行时索引反查设备。 |
| 3 | 序列号反查 | 如运行时保存序列号，可作为补充。 |

如果无法识别设备，仍可记录原始事件日志，但不写入业务表，避免设备字段不可信。

## 队列模型

建议内部使用 `RawAcsAlarmEvent`：

| 字段 | 说明 |
| --- | --- |
| `ReceivedAt` | 服务接收时间。 |
| `Command` | 原始命令号。 |
| `DeviceId` | 反查到的设备 ID。 |
| `DeviceIp` | 来源 IP。 |
| `DeviceName` | 运行时设备名称。 |
| `DeviceSerialNo` | 序列号或 IP。 |
| `AlarmInfoBytes` | 复制后的 ACS 报警结构字节。 |
| `PictureBytes` | 复制后的抓拍图片字节，可为空。 |
| `RawSummary` | 摘要日志信息。 |
| `RequestId` | 回调链路 ID。 |

## 队列容量与降级

| 场景 | 处理 |
| --- | --- |
| 队列正常 | 投递后台消费者。 |
| 队列接近满 | 记录 `Warn`，继续投递。 |
| 队列已满 | 丢弃当前事件并记录 `Error`，包含设备、命令号和接收时间。 |
| 服务停止 | 不再接收新事件，已入队事件按停止策略处理。 |

由于阶段 7 不新增事件暂存表，队列满时无法保证实时事件不丢。现场应通过队列容量、消费性能和日志监控降低风险。

## V50 与 V31 兼容

阶段 3 已预留 V50 优先、V31 兼容策略：

| 模式 | 阶段 7 处理 |
| --- | --- |
| V50 连续回调正常 | 直接使用 V50 回调投递原始事件。 |
| 现场要求 V31 确认 | 使用阶段 3.7 的 V31 兼容适配，阶段 7 仍只消费统一 `RawAcsAlarmEvent`。 |

阶段 7 不在业务处理里分叉 V50/V31；差异应被阶段 3 的回调适配层抹平。

## 不做的事

| 不做内容 | 原因 |
| --- | --- |
| 不解析 AIOP | AIOP 属于后续阶段。 |
| 不在回调线程入库 | 避免阻塞 SDK。 |
| 不持久化原始队列 | 阶段 7 不新增表。 |
| 不修改布防生命周期 | 布防编排属于阶段 4，阶段 7 只消费回调。 |

## 测试

| 测试 | 验证 |
| --- | --- |
| ACS 命令识别 | `COMM_ALARM_ACS` 被投递队列。 |
| 非 ACS 命令忽略 | 不进入 ACS 处理队列。 |
| IP 反查设备 | 能根据 `sDeviceIP` 找到设备。 |
| UserID 反查设备 | IP 为空时可反查。 |
| 队列满降级 | 不阻塞回调线程并记录错误。 |
| 回调异常保护 | 异常被捕获，不向 SDK 抛出。 |

# 阶段 8 / 任务 01：阶段边界与任务总览

## 目标

阶段 8 将前面阶段形成的能力整理为可交付、可验证、可部署、可运维的实施方案，包括自动化测试矩阵、构建命令、发布包结构、配置模板、Windows Service 安装运行、现场联调、排障和回滚。

阶段 8 不新增业务功能，不改变 gRPC 契约，不修改现有数据库表结构；它负责把阶段 1-7 和后续阶段的实施、验证、部署动作固化。

## 硬性边界

| 边界 | 要求 |
| --- | --- |
| 不改业务契约 | 不新增、不修改 gRPC 方法名、请求 JSON、响应 JSON、错误码。 |
| 不改表结构 | 阶段 8 不新增表，不修改现有表结构。 |
| 不引入硬件依赖单测 | 自动化单元测试和 mock 集成测试不依赖真实设备。 |
| 真实设备显式开启 | 现场设备联调必须通过配置和测试分类显式开启。 |
| 配置入口唯一 | 业务配置仍只读取运行目录 `Configuration/appsettings.json`。 |
| 服务名固定 | Windows Service 名称固定为 `ControlDoor`。 |
| 发布包可检查 | 发布包必须包含配置模板、SDK DLL 放置说明、日志/抓拍目录和运行前检查说明。 |

## 现实仓库状态说明

当前文档仓库中未必已经存在最终 `.sln/.csproj` 和发布脚本文件。阶段 8 固定的是实施时必须满足的命令和产物要求；具体工程文件落地后，应按本阶段文档执行并把实际脚本补齐。

## 阶段完成标准

| 标准 | 说明 |
| --- | --- |
| 自动化测试清楚 | 每一层测试目标、命令、依赖和是否需要硬件明确。 |
| 发布包清楚 | 现场拿到包后知道哪些文件必须存在、哪些目录可为空。 |
| 配置可验证 | `--validate-config` 能在启动前发现关键问题。 |
| 服务可运维 | 安装、启动、停止、卸载和日志路径明确。 |
| 联调可执行 | 真实设备联调有白名单、开关、步骤和通过标准。 |
| 回滚可执行 | 出问题时能退回旧服务包和旧配置。 |

# 门禁设备控制服务完整实施方案

## Summary

建设一个 Windows 后台服务 `ControlDoor`，用于统一管理海康门禁/人脸设备连接、权限/人员/人脸下发、离线补偿、人脸事件入库和设备管理 gRPC 接口。摄像头 AIOP 报警联动门禁常闭窗口纳入后续阶段规划，不作为首期核心范围。服务对外完全兼容既定 gRPC 方法名、请求 JSON、响应 JSON、错误码、数据库访问约束和配置文件路径。

核心运行模型采用“设备固定执行通道”：所有海康 SDK/ISAPI 设备操作都封装成任务，按 `device_id` 路由到固定串行通道，同一设备的登录、登出、状态检测、布防、远程配置、权限下发、门控常闭/恢复等操作按顺序执行；不同设备可以并行执行。SDK 回调线程只复制必要数据并投递后台任务，不直接执行耗时设备控制。

## 已确认边界

- 本文档目标是形成一份完整、详细、可直接实施的门禁系统方案；当前阶段先讨论并固化方案，不默认进入代码实施。
- 本方案以独立、完整、可落地的门禁系统实施方案为目标；后续文档统一使用“建设”“实现”“实施”“兼容”表述。
- gRPC 对外契约必须完全兼容：服务名、方法名、方法类型、请求 JSON、响应 JSON、错误码、鉴权约定和字符串传输方式均不得破坏既有调用方。
- 现有数据库表结构零变更：不得修改现有表的字段、类型、索引、约束和语义；如确需保存新业务状态，可以新增表，并必须通过 `database/` 脚本和相关文档同步。
- 摄像头 AIOP 报警联动门禁常闭属于后续阶段；首期方案优先完成门禁设备、权限人员人脸、离线补偿、人脸事件和设备管理能力。

## 总体架构

### 1. 服务分层与模块

- `Host`：Windows Service 入口，负责生命周期、配置加载、依赖装配、启动/停止顺序。
- `Configuration`：只读取运行目录 `Configuration/appsettings.json`；不从 `.exe.config` 读取业务配置。
- `Database`：使用 SQL Server；`devices`、`system_users`、`attendance_gate_v2`、`device_operation_retry_states`、`face_event_checkpoint` 等现有表结构零变更。
- `GrpcApi`：使用 `Grpc.Core` 手工注册 `string -> string` 和 `string -> stream string` 方法；请求/响应均为 UTF-8 JSON 字符串。
- `DeviceRuntime`：维护设备内存状态、连接信息、UserID、AlarmHandle、能力、状态、错误、反查索引。
- `DeviceSdkDispatcher`：按设备 ID 分配到固定串行执行通道，承载所有设备 SDK/ISAPI 调用。
- `HikvisionGateway`：封装 `HCNetSDK` P/Invoke、ISAPI XML/JSON 调用、错误码读取、技术规范校验。
- `PermissionSync`：权限、人员、人脸、删除、查询、采集业务。
- `RetryEngine`：离线设备写操作补偿，使用 `device_operation_retry_states` 记录补偿状态。
- `FaceEventIngestion`：门禁 ACS 人脸事件实时入库、离线事件补偿、断点维护。
- `CameraDoorInterlock`：后续阶段模块，负责摄像头 AIOP 报警联动门禁常闭窗口。
- `Observability`：结构化日志、敏感字段脱敏、操作链路 ID、后台任务状态日志。

### 2. 设备固定执行通道

- 启动时创建固定数量的 `DeviceSdkWorker`，每个 worker 一个 FIFO 队列。
- 设备任务路由规则固定为 `workerIndex = Abs(deviceId) % workerCount`。
- 同一 `device_id` 的所有设备操作必须进入同一 worker 队列；禁止业务模块直接调用 `HCNetSDK` 或直接发 ISAPI。
- 每个任务包含：`DeviceId`、`OperationName`、`Priority`、`TimeoutMs`、`CancellationToken`、`ExecuteAsync`、`RequestId`。
- 高优先级任务顺序：登出/删除、门禁恢复、门禁常闭、报警布防、权限/人员/人脸写操作、状态检测。
- 任务执行期间由 worker 读取设备当前 `UserID`、连接状态和配置；任务完成后统一写回设备状态和错误信息。
- 失败后立即读取 `NET_DVR_GetLastError`，在同一任务上下文中记录日志，避免错误码串扰。
- 超时控制在任务外层完成；SDK 调用无法强制中断时记录“任务执行超时风险”，后续任务继续按队列策略处理。
- 恢复重试不得在 worker 中 `Thread.Sleep`；失败后由定时器/延迟调度重新入队。

### 3. 设备生命周期

- 启动时从 `dbo.devices` 加载启用设备，建立 `device_id`、`ip_address`、`sdkUserId` 反查索引。
- 登录任务调用 `NET_DVR_Login_V40`，成功后更新 `UserID`、序列号、能力、状态和 `last_used_time`。
- 登出任务调用 `NET_DVR_Logout_V30`，并清理 `UserID`、AlarmHandle、远程配置句柄、反查索引。
- 状态检测任务按 `DeviceConnection.StatusCheckIntervalMs` 调度，通过 SDK 工作状态和能力接口判断在线、离线、门状态。
- 重连策略按配置执行指数退避、最大次数、永久失败冷却；重连任务也必须进入设备通道。
- 设备管理 gRPC 的新增、删除、断开、重连、状态查询逻辑保持 JSON 契约不变；写库仍操作 `dbo.devices`。

### 4. gRPC 外部接口

保持以下服务名、方法名、方法类型、JSON 字符串传输方式不变：

- `permission.PermissionSyncService/SyncPermissions`
- `permission.PermissionSyncService/SyncPersons`
- `permission.PermissionSyncService/DeleteFaces`
- `permission.PermissionSyncService/DeletePersons`
- `permission.PermissionSyncService/GetFaces`
- `permission.PermissionSyncService/GetEnrollmentStatus`
- `permission.PermissionSyncService/CaptureFaceStream`
- `device.AccessControlService/GetDeviceStatus`
- `device.AccessControlService/AddDevice`
- `device.AccessControlService/DeleteDevice`
- `device.AccessControlService/DisconnectDevice`
- `device.AccessControlService/ReconnectDevice`

统一响应结构保持：

```json
{
  "requestId": "请求ID",
  "success": true,
  "code": "OK",
  "message": "处理结果说明",
  "errors": [],
  "errorDetails": []
}
```

错误码保持：`OK`、`PARTIAL_SUCCESS`、`FAILED`、`INVALID_ARGUMENT`、`BATCH_TOO_LARGE`、`NOT_FOUND`、`INTERNAL_ERROR`、`UNAUTHENTICATED`、`DEVICE_ERROR`、`DB_ERROR`、`SDK_ERROR`、`FACE_TOO_LARGE`。

### 5. 权限、人员、人脸业务

- `SyncPermissions`：解析员工编号和权限编号，遍历目标在线设备，向设备下发权限；离线或可重试失败写入 `device_operation_retry_states`。
- `SyncPersons`：解析人员基础信息、有效期、启用状态、人脸 Base64；先下发人员，再按需要下发人脸。
- `DeleteFaces`：删除指定员工人脸；失败按设备和员工写入补偿状态。
- `DeletePersons`：删除指定员工人员信息；失败按设备和员工写入补偿状态。
- `GetFaces`：查询指定员工在设备端的人脸信息，返回每个员工/设备的结果明细。
- `CaptureFaceStream`：从录入设备采集人脸，返回一帧流式 JSON 结果；任务状态写入内存任务存储。
- 人脸图片限制 200KB；日志必须脱敏 `faceImageBase64`、`password`、`apiKey` 等敏感字段。
- 所有设备写操作进入设备固定执行通道；离线补偿重试也进入相同通道。

### 6. 离线补偿业务

- 保持 `device_operation_retry_states` 表结构、唯一约束和索引不变。
- 后台按 `DeviceOperationRetry.ScanIntervalSeconds` 扫描 `next_retry_at` 到期任务。
- 补偿任务按 `device_id` 投递到对应设备执行通道。
- 成功后清理对应 pending 标记；失败后更新 `attempt_count`、`next_retry_at`、`last_error`。
- 达到 `MaxRetryAttempts` 后写入 `exhausted_at`。
- 清理终态失败记录按 `FailureRetentionDays` 执行。
- 对同一设备同一员工只保留一条补偿状态，并合并权限、人员、人脸、删除动作。

### 7. 人脸事件入库

- 启动时注册 SDK 报警回调，布防任务进入设备固定执行通道。
- SDK 回调收到 `COMM_ALARM_ACS` 后只复制必要字段、图片指针数据、设备来源信息，然后投递事件处理队列。
- 实时事件解析员工编号、事件时间、设备、序列号、抓拍、认证结果，写入 `attendance_gate_v2`。
- 使用业务流水 `id` 唯一约束避免重复入库。
- 抓拍图片保存到 `FaceEventLogging.SnapshotRootDirectory` 或默认 `snapshots` 目录，表中保存路径。
- 设备恢复在线后，根据 `face_event_checkpoint` 拉取历史事件补偿；补偿期间实时事件进入缓冲，补偿完成后释放缓冲。
- `FaceEventLogging.ExcludedDeviceIds/ExcludedDeviceIps` 只排除人脸事件入库和补偿，不影响被摄像头联动配置引用的 AIOP 报警布防。

### 8. 后续阶段：摄像头 AIOP 报警联动门禁常闭

- 配置使用顶层 `CameraAlarmDoorInterlock`：
  - `Enabled`
  - `WindowSeconds`
  - `DoorControlSdkLockTimeoutMs` 可保留为兼容配置，但设备通道方案中不再作为互斥等待参数使用。
  - `RestoreRetryAttempts`
  - `RestoreRetryIntervalMs`
  - `Mappings`
- 摄像头和门禁目标识别规则固定：优先 `Ip`；只有 `Ip` 为空时使用 `Id`；`DoorNos` 省略默认 `[1]`。
- SDK 回调收到 `0x4021 / COMM_UPLOAD_AIOP_VIDEO` 后识别来源摄像头：
  - 优先 `NET_DVR_ALARMER.sDeviceIP`
  - IP 为空时用 `lUserID` 反查设备
- 只要来源命中配置摄像头，就触发联动；AIOP JSON 解析失败不阻断。
- 每个摄像头维护独立窗口：首次报警开启 `now + WindowSeconds`；窗口内重复报警只记日志，不刷新窗口，不重复下发常闭。
- 一个摄像头映射多个门禁设备/门号时，全部投递常闭任务。
- 多个摄像头影响同一个门时，门目标维护活动摄像头集合；最后一个摄像头窗口结束后才投递恢复任务。
- 常闭和恢复通过设备固定执行通道调用 `NET_DVR_ControlGateway`：
  - 常闭：`NET_DVR_GATEWAY_CONTROL_ALWAYS_CLOSE`
  - 恢复：`NET_DVR_GATEWAY_CONTROL_CLOSE`
- 恢复失败按配置延迟重新投递，不阻塞 worker。
- 服务停止时对所有仍处于活动状态的门目标投递 best-effort 恢复任务，并记录结果。

### 9. 配置与部署

- 配置文件路径固定为运行目录 `Configuration/appsettings.json`。
- `Service.GrpcListenPort` 默认 5001，使用 `ServerCredentials.Insecure`。
- `Service.GrpcManagementApiKey` 仅保护 `device.AccessControlService`；为空时不强制鉴权但记录警告。
- 日志目录、保留天数、gRPC payload 日志、脱敏策略按配置执行。
- 项目输出生成 Windows Service 可执行文件，服务名固定为 `ControlDoor`。
- Build 后产出 `bin` 和 `门禁publish\ServicePackage` 发布包。
- 数据库连接、命令超时、设备连接、重连、补偿、人脸事件、摄像头联动配置项保持兼容；现有表结构不得修改。

## 已确认假设与约束

- 外部 gRPC 契约、配置路径、Windows 服务名均保持不变。
- 现有数据库表结构零变更；新增业务状态允许新增表，但不能修改现有表。
- 海康 SDK 允许不同设备并行调用；同一设备所有 SDK/ISAPI 调用必须串行。
- 所有设备操作必须通过 `DeviceSdkDispatcher`，业务模块不得直接调用 `HCNetSDK` 或直接访问设备 ISAPI。
- 摄像头报警联动为后续阶段能力；默认不新增数据库表，映射关系只存在 JSON 配置，除非后续讨论确认需要持久化联动状态。
- “长臂状态”按“常闭/禁止入门”实现，窗口结束恢复为普通受控状态，不恢复为常开。
- AIOP 报警不按 `type` 过滤；配置摄像头收到 `0x4021` 即联动。
- `DoorControlSdkLockTimeoutMs` 作为兼容配置保留；设备通道模型以任务超时、队列优先级和延迟重试控制设备操作。
- 技术实现以本地 `设备网络SDK编程指南（明眸-以人为中心）`、仓库 SDK 缓存文档和 `HCNetSDK` 头文件/声明为准。

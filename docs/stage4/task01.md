# 阶段 4 / 任务 01：阶段边界与任务总览

## 目标

阶段 4 的目标是把阶段 2 的设备固定执行通道、阶段 3 的海康 SDK 网关接入实际设备生命周期，并实现 `device.AccessControlService` 的 5 个设备管理 gRPC 方法。

阶段 4 完成后，服务应能从 `Configuration/devices.json` 加载启用设备，按设备固定通道执行登录、登出、状态检测、重连、布防和撤防，并能通过既定 gRPC 契约查询、新增、删除、断开和重连设备。

## 硬性边界

| 边界 | 要求 |
| --- | --- |
| gRPC 完全兼容 | 服务名、方法名、方法类型、请求 JSON、响应 JSON、错误码和 `x-api-key` 约定保持不变。 |
| 设备清单 JSON 化 | 当前版本设备主数据只读写 `Configuration/devices.json`。 |
| 设备操作通道化 | 登录、登出、状态检测、布防、撤防、重连必须进入 `DeviceSdkDispatcher`。 |
| 不做权限/人员/人脸业务 | 权限、人员、人脸同步属于阶段 5。 |
| 不写离线补偿 | 设备生命周期失败只更新内存状态和日志；写补偿属于阶段 6。 |
| 不做事件入库 | 报警回调原始事件可投递，ACS 解析和入库属于阶段 7。 |
| 不做摄像头联动 | AIOP 联动门禁常闭属于后续阶段。 |

## 实施顺序

1. 实现设备记录读取模型和运行时注册，保证不连接设备也能形成内存快照。
2. 接入登录/登出任务，成功后维护 UserID、IP、AlarmHandle 反查索引。
3. 增加状态检测和重连调度，失败不阻塞其他设备。
4. 接入布防/撤防编排，保持事件解析和入库后置。
5. 实现设备管理 gRPC handler，保持 JSON 契约完全兼容。
6. 补齐阶段 4 mock 测试、数据库兼容测试和手动联调说明。

## 通过标准

| 标准 | 说明 |
| --- | --- |
| 设备可加载 | 启动时能读取 `enabled=true` 的 `Configuration/devices.json` 记录并注册内存运行时。 |
| 登录可编排 | 登录成功更新运行时 UserID 和状态，登录失败进入离线/重连策略。 |
| 清理可控 | 删除、断开、服务停止时按撤防、登出、索引清理顺序执行。 |
| gRPC 可用 | 5 个设备管理方法均按 `docs/gRPC接口清单.md` 的契约返回。 |
| 设备清单兼容 | 不改 gRPC 契约，新增/删除设备写回 JSON 清单。 |
| 测试可证明 | mock gateway 可覆盖成功、失败、超时、重连、布防失败等关键路径。 |

# 底层数据库文档

本文档说明当前门禁服务使用的 SQL Server 数据库、脚本、核心表结构和代码访问关系。

## 1. 数据库概览

| 项目 | 内容 |
| --- | --- |
| 数据库类型 | SQL Server |
| 默认数据库名 | `ruoyi-vue-pro` |
| 默认排序规则 | `Chinese_PRC_CI_AS` |
| 连接配置来源 | 运行目录 `Configuration\appsettings.json` 的 `Database.ConnectionString` |
| 命令超时配置 | `Database.CommandTimeoutSeconds`，默认 30 秒 |
| 访问封装 | `SqlServerDatabase.cs` |

生产部署注意：示例配置和初始化脚本中包含默认账号与示例密码，正式环境必须替换，并限制数据库账号权限。

## 2. 数据库脚本清单

| 脚本 | 用途 | 是否可重复执行 | 说明 |
| --- | --- | --- | --- |
| `Database/初始化_门禁设备系统用户与数据库登录.sql` | 初始化数据库、人员表、数据库登录与用户 | 否 | 会删除并重建 `ruoyi-vue-pro` 数据库；设备主数据来自 `Configuration/devices.json` |
| `Database/专项_20260105_门禁进出记录表V2.sql` | 创建门禁进出记录表 `attendance_gate_v2` | 基本可重复 | 表不存在时才创建 |
| `Database/专项_20260309_设备操作重试状态表.sql` | 创建设备操作补偿表 `device_operation_retry_states` 并补齐索引 | 可重复 | 包含字段存在性与索引存在性判断 |
| `Database/专项_人脸事件日志与断点表.sql` | 历史版本的人脸事件补偿检查点表 `face_event_checkpoint` | 可重复 | 阶段 7 当前方案不读写，仅保持旧库兼容 |

建议执行顺序：

1. `初始化_门禁设备系统用户与数据库登录.sql`
2. `专项_20260105_门禁进出记录表V2.sql`
3. `专项_20260309_设备操作重试状态表.sql`
4. `专项_人脸事件日志与断点表.sql`

## 3. 连接与访问方式

### 3.1 配置入口

服务启动后通过 `ExternalConfiguration` 读取运行目录下的：

```text
Configuration\appsettings.json
```

数据库相关配置：

```json
{
  "Database": {
    "ConnectionString": "SQL Server连接字符串",
    "CommandTimeoutSeconds": 30
  }
}
```

### 3.2 访问封装

`SqlServerDatabase` 提供以下能力：

| 能力 | 说明 |
| --- | --- |
| 建立连接 | 使用 `SqlConnection` 打开 SQL Server 连接 |
| 连接重试 | 默认最多重试 10 次，间隔 60 秒 |
| 命令创建 | `CreateCommand(sql)` 自动设置命令超时时间 |
| 错误记录 | 保存 `ErrorMessage` 和 `ErrorNumber`，并写服务日志 |
| 资源释放 | `Disconnect` / `Dispose` 关闭连接并释放资源 |

## 4. 核心表：`dbo.system_users`

### 4.1 用途

统一账号体系人员信息表，额外增加门禁权限同步字段。权限同步逻辑会根据员工编号更新门禁相关字段。

### 4.2 表结构

| 字段 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `id` | `BIGINT IDENTITY(1,1)` | 是 | 自增 | 人员主键 |
| `username` | `NVARCHAR(30)` | 是 | 无 | 用户名/员工编号，唯一 |
| `password` | `NVARCHAR(100)` | 是 | 空字符串 | 账号密码字段 |
| `nickname` | `NVARCHAR(30)` | 是 | 无 | 昵称/姓名 |
| `remark` | `NVARCHAR(500)` | 否 | NULL | 备注 |
| `dept_id` | `BIGINT` | 否 | NULL | 部门 ID |
| `post_ids` | `NVARCHAR(255)` | 否 | NULL | 岗位 ID 集合 |
| `email` | `NVARCHAR(50)` | 否 | 空字符串 | 邮箱 |
| `mobile` | `NVARCHAR(11)` | 否 | 空字符串 | 手机号 |
| `sex` | `TINYINT` | 否 | `0` | 性别 |
| `avatar` | `NVARCHAR(512)` | 否 | 空字符串 | 头像 |
| `status` | `TINYINT` | 是 | `0` | 账号状态 |
| `access_permission` | `TINYINT` | 是 | `2` | 门禁权限等级 |
| `last_synced_level` | `TINYINT` | 否 | NULL | 最近已同步权限等级 |
| `permission_updated_at` | `DATETIME2(3)` | 否 | NULL | 权限更新时间 |
| `last_synced_at` | `DATETIME2(3)` | 否 | NULL | 最近同步时间 |
| `login_ip` | `NVARCHAR(50)` | 否 | 空字符串 | 最近登录 IP |
| `login_date` | `DATETIME2(3)` | 否 | NULL | 最近登录时间 |
| `creator` | `NVARCHAR(64)` | 否 | 空字符串 | 创建人 |
| `create_time` | `DATETIME2(3)` | 是 | `SYSDATETIME()` | 创建时间 |
| `updater` | `NVARCHAR(64)` | 否 | 空字符串 | 更新人 |

| `tenant_id` | `BIGINT` | 是 | `1` | 租户 ID |

| `idx_system_users_status` | 普通索引 | 加速按状态筛选 |
| `idx_system_users_permission` | 普通索引 | 加速按门禁权限筛选 |

### 4.4 代码使用点

| 操作 | 代码位置 | 说明 |
| --- | --- | --- |
| 权限同步状态更新 | `DeviceOperationRetryStore`、`PermissionRefreshManager` | 更新 `access_permission`、`last_synced_level`、`permission_updated_at`、`last_synced_at` 等字段 |

## 6. 核心表：`dbo.device_operation_retry_states`

### 6.1 用途

设备离线写操作补偿表。权限、人员、人脸下发/删除在设备离线或临时失败时，会将待执行操作记录到此表，后台补偿任务后续重试。

### 6.2 表结构

| 字段 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `id` | `BIGINT IDENTITY(1,1)` | 是 | 自增 | 主键 |
| `device_id` | `INT` | 是 | 无 | 目标设备 ID |
| `employee_id` | `NVARCHAR(64)` | 是 | 无 | 员工编号 |
| `permission_level` | `INT` | 否 | NULL | 待同步权限等级 |
| `permission_payload` | `NVARCHAR(MAX)` | 否 | NULL | 权限补偿 JSON 载荷，保存员工号、权限等级和可选姓名；旧数据为空时补偿使用员工号兜底 |
| `permission_pending` | `BIT` | 是 | `0` | 是否存在权限同步待补偿 |
| `permission_sync_completion_blocked` | `BIT` | 是 | `1` | 是否阻塞权限同步完成标记 |
| `person_payload` | `NVARCHAR(MAX)` | 否 | NULL | 人员下发 JSON 载荷 |
| `person_pending` | `BIT` | 是 | `0` | 是否存在人员下发待补偿 |
| `face_payload` | `NVARCHAR(MAX)` | 否 | NULL | 人脸下发 JSON 载荷 |
| `face_pending` | `BIT` | 是 | `0` | 是否存在人脸下发待补偿 |
| `delete_person_pending` | `BIT` | 是 | `0` | 是否存在删除人员待补偿 |
| `delete_face_pending` | `BIT` | 是 | `0` | 是否存在删除人脸待补偿 |
| `attempt_count` | `INT` | 是 | `0` | 已重试次数 |
| `next_retry_at` | `DATETIME2(0)` | 否 | NULL | 下次重试时间 |
| `last_error` | `NVARCHAR(2000)` | 否 | NULL | 最近错误 |
| `last_attempt_at` | `DATETIME2(0)` | 否 | NULL | 最近尝试时间 |
| `exhausted_at` | `DATETIME2(0)` | 否 | NULL | 达到最大重试或终止时间 |
| `created_at` | `DATETIME2(0)` | 是 | `SYSDATETIME()` | 创建时间 |
| `updated_at` | `DATETIME2(0)` | 是 | `SYSDATETIME()` | 更新时间 |

### 6.3 约束与索引

| 名称 | 类型 | 字段 | 说明 |
| --- | --- | --- | --- |
| `PK_device_operation_retry_states` | 主键 | `id` | 表主键 |
| `UQ_device_operation_retry_states_device_employee` | 唯一索引 | `device_id`, `employee_id` | 同一设备同一员工只保留一条补偿状态 |
| `IX_device_operation_retry_states_next_retry_at` | 过滤索引 | `next_retry_at` INCLUDE `device_id`, `employee_id`, `updated_at` | 加速扫描未终止的到期补偿任务 |
| `IX_device_operation_retry_states_employee_permission` | 普通索引 | `employee_id`, `permission_pending` | 加速按员工和权限状态查询 |
| `IX_device_operation_retry_states_exhausted_at` | 普通索引 | `exhausted_at` INCLUDE `device_id`, `employee_id`, `updated_at` | 加速清理或查看终止任务 |

### 6.4 代码使用点

| 操作 | 代码位置 | 说明 |
| --- | --- | --- |
| 写入/更新补偿状态 | `DeviceOperationRetryStore` | 保存权限、人员、人脸、删除动作的待补偿状态 |
| 扫描到期任务 | `DeviceOperationRetryManager` | 按 `next_retry_at` 加载待处理任务 |
| 重试排程 | `DeviceOperationRetryStore.ScheduleRetry` | 更新 `attempt_count`、`next_retry_at`、`last_error` |
| 终止失败 | `DeviceOperationRetryStore.MarkTerminalFailure` | 写入 `exhausted_at` |
| 清理过期失败 | `DeviceOperationRetryStore.CleanupExpiredFailures` | 按保留天数删除终止记录 |

## 8. 数据流关系

### 8.1 服务启动数据流

1. 读取 `Configuration\appsettings.json`。
2. 使用 `Database.ConnectionString` 建立 SQL Server 连接。
3. 从 `Configuration\devices.json` 加载所有设备。
4. 启动设备状态监控和重连策略。
5. 启动人脸事件订阅，事件写入 `attendance_gate_v2`。
6. 启动离线补偿任务，扫描 `device_operation_retry_states`。
7. 启动 gRPC 服务，对外接收同步、采集和设备管理请求。

### 8.2 权限/人员/人脸下发数据流

1. gRPC 接收 JSON 请求。
2. 解析员工编号、权限、人脸或人员信息。
3. 遍历在线目标设备并调用海康设备接口。
4. 在线成功则返回成功明细。
5. 设备离线或可重试失败时写入 `device_operation_retry_states`。
6. 后台补偿任务在设备恢复或到期扫描时再次执行。

### 8.3 人脸事件入库数据流

1. `FaceEventService` 通过 SDK 接收设备事件。
2. 解析员工编号、设备、事件时间、方向、抓拍等信息。
3. 抓拍图片保存到配置的图片目录。
4. 事件写入 `attendance_gate_v2`。
5. 设备离线恢复后，阶段 7 当前方案依赖客户端布防上传 `byCurrentEvent=2` 的离线事件，不再根据 `face_event_checkpoint` 主动拉取历史事件。

## 9. 运维注意事项

| 事项 | 建议 |
| --- | --- |
| 初始化脚本 | 不要在已有生产数据的库上直接执行初始化脚本，因为它会删除并重建数据库 |
| 默认凭据 | 部署前必须替换数据库登录密码、设备默认密码和管理 API Key |
| 表结构变更 | 建议后续统一改为版本化迁移脚本，避免初始化脚本和专项脚本分叉 |
| 索引维护 | `attendance_gate_v2` 是流水表，数据量增长后需要关注日期/人员查询索引和归档策略 |
| 补偿表清理 | 关注 `DeviceOperationRetry.FailureRetentionDays`，避免终止失败记录无限增长 |
| 图片目录 | 确保 `FaceEventLogging.SnapshotRootDirectory` 所在磁盘空间和权限充足 |
| 备份 | 至少备份 `Configuration\devices.json`、`system_users`、`attendance_gate_v2`、`device_operation_retry_states` |

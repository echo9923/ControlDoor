# 阶段 7：ACS 人脸事件入库与离线事件上传补偿 — 实施目标与任务清单

## 概述

阶段 7 的目标是实现门禁 ACS 人脸事件入库与离线事件上传补偿。服务通过阶段 3/4 已建立的报警回调和客户端布防能力接收 `COMM_ALARM_ACS`，解析 `NET_DVR_ACS_ALARM_INFO` 相关字段，保存抓拍图片，将实时事件和设备恢复后上传的离线事件写入现有 `dbo.attendance_gate_v2`。

阶段 7 只处理门禁/人脸设备 ACS 事件，不处理摄像头 AIOP 短衣短裤报警联动门禁常闭。

## 硬性边界

| 边界 | 要求 |
| --- | --- |
| 现有表零结构变更 | 不修改 `attendance_gate_v2` 和其他现有表结构。 |
| 不新增阶段 7 表 | 实时事件和离线事件上传补偿只使用 SDK 回调队列与现有事件表。 |
| 单一补偿方式 | 阶段 7 只采用客户端布防下的设备离线事件上传，识别 `byCurrentEvent == 2`。 |
| 不主动拉历史 | 不实现 `QueryHistoryEvents`，不使用 `NET_DVR_GET_ACS_EVENT` 主动查询历史事件，不读写 `face_event_checkpoint`。 |
| 回调线程轻量 | SDK 回调线程只复制必要数据并投递队列，不做数据库写入和复杂解析。 |
| 防重复依赖业务流水 | 使用 `attendance_gate_v2.id` 唯一语义避免重复入库。 |
| 不改变 gRPC 契约 | 阶段 7 不新增、不修改外部 gRPC 方法。 |
| 不做 AIOP 联动 | 摄像头 `COMM_UPLOAD_AIOP_VIDEO / 0x4021` 属于后续阶段。 |

## 任务 7.2：回调接收与事件队列

### 目标

在阶段 3 报警布防基础上实现 ACS 回调接收和队列投递。SDK 回调线程必须尽快返回，只负责复制必要数据、识别来源设备并把原始事件投递到后台队列。

### 回调线程规则

| 规则 | 说明 |
| --- | --- |
| 只复制 | 复制报警结构原始字节、图片字节和来源信息。 |
| 不写库 | 不访问 `attendance_gate_v2`。 |
| 不写文件 | 不保存抓拍图片。 |
| 不阻塞 | 队列满时按降级策略处理，不能长时间等待。 |
| 不抛异常 | 捕获所有异常并记录，避免影响 SDK 回调链路。 |

### 设备来源识别

优先级：

| 优先级 | 来源 | 说明 |
| --- | --- | --- |
| 1 | `NET_DVR_ALARMER.sDeviceIP` | 首选设备上报 IP。 |
| 2 | `lUserID` 反查 | IP 为空时用阶段 4 运行时索引反查设备。 |
| 3 | 序列号反查 | 如运行时保存序列号，可作为补充。 |

如果无法识别设备，仍可记录原始事件日志，但不写入业务表。

### 队列模型

建议内部使用 `RawAcsAlarmEvent`：

| 字段 | 说明 |
| --- | --- |
| `ReceivedAt` | 服务接收时间。 |
| `Command` | 原始命令号。 |
| `DeviceId` | 反查到的设备 ID。 |
| `DeviceIp` | 来源 IP。 |
| `DeviceName` | 运行时设备名称。 |
| `DeviceSerialNo` | 序列号或 IP。 |
| `AlarmInfoBytes` | 复制后的 ACS 报警结构字节。 |
| `PictureBytes` | 复制后的抓拍图片字节，可为空。 |
| `CurrentEventFlag` | `byCurrentEvent` 原值；`2` 表示设备上传的离线事件。 |
| `Source` | `Realtime` 或 `OfflineUpload`。 |
| `RawSummary` | 摘要日志信息。 |
| `RequestId` | 回调链路 ID。 |

### 队列容量与降级

| 场景 | 处理 |
| --- | --- |
| 队列正常 | 投递后台消费者。 |
| 队列接近满 | 记录 `Warn`，继续投递。 |
| 队列已满 | 丢弃当前事件并记录 `Error`，包含设备、命令号和接收时间。 |
| 服务停止 | 不再接收新事件，已入队事件按停止策略处理。 |

### V50 与 V31 兼容

阶段 3 已预留 V50 优先、V31 兼容策略。阶段 7 不在业务处理里分叉 V50/V31；差异应被阶段 3 的回调适配层抹平。

### 验收标准

- [ ] ACS 命令识别：`COMM_ALARM_ACS` 被投递队列。
- [ ] 非 ACS 命令忽略，不进入 ACS 处理队列。
- [ ] 能根据 `sDeviceIP` 或 `lUserID` 反查设备。
- [ ] 队列满时不阻塞回调线程并记录错误。
- [ ] 回调异常被捕获，不向 SDK 抛出。

## 任务 7.3：ACS 事件解析与标准模型

### 目标

把 `RawAcsAlarmEvent` 解析为标准 `AcsFaceEvent`，提取员工编号、事件时间、业务流水、方向、设备、卡号、认证结果、事件类型和原始载荷摘要。

### 标准模型

建议内部使用 `AcsFaceEvent`：

| 字段 | 说明 |
| --- | --- |
| `EventId` | 写入 `attendance_gate_v2.id` 的业务流水。 |
| `EmployeeId` | 员工编号，写入 `username`。 |
| `Nickname` | 从 `system_users` 查询补全，可为空。 |
| `EventTime` | 事件时间，写入 `record_datetime`。 |
| `RecordDate` | 事件日期。 |
| `RecordTime` | 事件时间。 |
| `Direction` | 进出方向，无法识别时按既有默认值 1。 |
| `DeviceName` | 设备名称。 |
| `DeviceSn` | 设备序列号、IP 或运行时标识。 |
| `CardNo` | 卡号，可为空。 |
| `EventType` | 事件类型。 |
| `AuthResult` | 认证结果，用于 `process_message` 或 raw payload。 |
| `PictureBytes` | 抓拍图片。 |
| `RawPayload` | 原始结构摘要或 JSON。 |
| `Source` | `Realtime` 或 `OfflineUpload`。 |

### EventId 生成规则

优先使用设备提供的稳定业务流水，保证离线上传事件和实时回调能得到同一个 `id`。

| 要求 | 说明 |
| --- | --- |
| 确定性 | 同一设备同一事件重复解析得到同一个值。 |
| 低碰撞 | 包含设备标识、事件时间、员工/卡号、事件类型。 |
| long 范围 | 结果必须适配 `BIGINT`。 |

只有在设备没有可用流水时才使用哈希兜底，并在日志中标记 `eventIdGenerated=true`。

### 事件过滤

| 场景 | 处理 |
| --- | --- |
| 员工编号为空且卡号为空 | 不入库，记录 `Warn`。 |
| 事件时间无效 | 使用接收时间兜底，并在 raw payload 说明。 |
| 设备无法识别 | 不入库，记录 `Warn`。 |
| 非门禁 ACS 事件 | 可记录 debug，不写业务表。 |
| 认证失败事件 | 可入库，`process_message` 标记失败原因。 |

### 昵称补全

| 步骤 | 动作 |
| --- | --- |
| 1 | 用 `EmployeeId` 查询或缓存 `system_users`。 |
| 2 | 找到昵称则填入 `nickname`。 |
| 3 | 找不到则 `nickname` 为空，不阻塞入库。 |

### raw payload

`attendance_gate_v2.raw_payload` 为 `VARCHAR(MAX)`，建议保存 UTF-8 JSON 字符串，包含：

| 字段 | 说明 |
| --- | --- |
| `source` | `Realtime` / `OfflineUpload`。 |
| `deviceId` | 设备 ID。 |
| `deviceIp` | 设备 IP。 |
| `eventType` | 事件类型。 |
| `authResult` | 认证结果。 |
| `sdkCommand` | 命令号。 |
| `byCurrentEvent` | ACS 扩展字段原值。 |
| `eventIdGenerated` | 是否使用兜底 ID。 |

### 验收标准

- [ ] 标准 ACS 成功事件字段映射完整。
- [ ] 认证失败事件可入库并标记失败。
- [ ] 员工为空时不入库并记录原因。
- [ ] EventId 稳定：同一输入重复解析得到同一 ID。
- [ ] 昵称补全：找到用户时写 nickname，找不到不失败。

## 任务 7.4：抓拍图片保存与 raw payload

### 目标

实现抓拍图片保存和原始载荷记录策略。把 ACS 事件中的图片保存到 `FaceEventLogging.SnapshotRootDirectory`，并将路径写入 `attendance_gate_v2.snapshot_path`。

### 目录规则

| 项目 | 规则 |
| --- | --- |
| 根目录 | `FaceEventLogging.SnapshotRootDirectory`，默认 `snapshots`。 |
| 相对路径 | 相对运行目录解析。 |
| 不存在 | 模块启动时自动创建。 |
| 创建失败 | 人脸事件模块启动失败或进入不可用状态。 |
| 文件分层 | 建议按 `yyyyMMdd/deviceId/` 分目录。 |

### 文件命名

建议格式：

`	ext
{eventTime:yyyyMMddHHmmssfff}_{deviceId}_{employeeId}_{eventId}.jpg
`

| 要求 | 说明 |
| --- | --- |
| 文件名安全 | 过滤员工编号中的非法路径字符。 |
| 可追踪 | 包含时间、设备、员工、事件 ID。 |
| 防覆盖 | 同名存在时可追加短后缀，但数据库 `id` 仍负责业务防重。 |

### 图片写入流程

| 步骤 | 动作 |
| --- | --- |
| 1 | 判断事件是否包含图片字节。 |
| 2 | 无图片则 `snapshot_path = NULL`，不影响入库。 |
| 3 | 校验图片大小和基础格式。 |
| 4 | 生成目录和文件名。 |
| 5 | 写入临时文件。 |
| 6 | 写入完成后原子移动到目标文件。 |
| 7 | 返回用于入库的路径。 |

### 路径保存

`snapshot_path` 长度为 `NVARCHAR(255)`。保存前必须检查长度：

| 场景 | 处理 |
| --- | --- |
| 路径长度 <= 255 | 正常写入。 |
| 路径长度 > 255 | 使用较短文件名或相对路径。 |
| 仍超过 255 | 不写图片路径，raw payload 记录原因。 |

### 文件写入失败

| 场景 | 处理 |
| --- | --- |
| 磁盘不可写 | 记录 `Error`，事件仍可入库但 `snapshot_path = NULL`。 |
| 图片字节为空 | 正常入库，路径为空。 |
| 图片格式未知 | 默认不保存并记录。 |
| 同名冲突 | 追加短后缀。 |

抓拍保存失败不应阻塞事件入库，除非模块启动时根目录不可创建。

### 验收标准

- [ ] 目录自动创建：根目录不存在时可创建。
- [ ] 图片保存成功：文件存在且路径入库。
- [ ] 无图片事件：入库成功，路径为空。
- [ ] 路径超长：自动缩短或记录原因。
- [ ] 写文件失败：事件仍可入库，日志有错误。
- [ ] 文件名非法字符被安全替换。

## 任务 7.5：attendance_gate_v2 入库与防重复

### 目标

实现 `FaceEventRepository.InsertEvent`，把标准 `AcsFaceEvent` 写入现有 `dbo.attendance_gate_v2` 表，并依赖 `id` 唯一索引保证实时事件和离线上传事件不会重复入库。

### 字段映射

| 表字段 | 来源 |
| --- | --- |
| `id` | `AcsFaceEvent.EventId`。 |
| `username` | `AcsFaceEvent.EmployeeId`。 |
| `nickname` | `AcsFaceEvent.Nickname`。 |
| `record_datetime` | `AcsFaceEvent.EventTime`。 |
| `record_date` | `EventTime.Date`。 |
| `record_time` | `EventTime.TimeOfDay`。 |
| `direction` | `AcsFaceEvent.Direction`。 |
| `device_name` | `AcsFaceEvent.DeviceName`。 |
| `device_sn` | `AcsFaceEvent.DeviceSn`。 |
| `card_no` | `AcsFaceEvent.CardNo`。 |
| `snapshot_path` | `SnapshotStorage` 返回路径。 |
| `raw_payload` | 标准 raw payload JSON。 |
| `event_type` | `AcsFaceEvent.EventType`。 |
| `process_status` | 默认 0。 |
| `process_message` | 认证结果或处理说明。 |
| `processed_at` | 可为空。 |
| `creator` | 固定服务标识，例如 `ControlDoor`。 |
| `create_time` | 当前时间或事件入库时间。 |
| `updater` | 固定服务标识。 |
| `update_time` | 当前时间。 |
| `deleted` | 固定 `0`。 |
| `tenant_id` | 默认 1，除非现有配置另有明确值。 |

### 入库流程

| 步骤 | 动作 |
| --- | --- |
| 1 | 校验 `EventId`、`EmployeeId`、`EventTime`。 |
| 2 | 补全昵称。 |
| 3 | 保存抓拍图片并得到路径。 |
| 4 | 构建 insert 命令。 |
| 5 | 执行插入。 |
| 6 | 捕获唯一键冲突，识别为重复事件。 |
| 7 | 返回 `Inserted` 或 `Duplicate` 结果。 |

### 防重复规则

| 场景 | 处理 |
| --- | --- |
| 实时事件重复回调 | 第二次插入命中 `ux_gate_v2_id`，返回 Duplicate。 |
| 离线上传事件补到实时已入库事件 | 命中唯一键，视为成功处理。 |
| 设备重复上传同一离线事件 | 命中唯一键，视为成功处理。 |
| EventId 生成碰撞 | 记录 `Error`，人工排查；实现上仍遵守唯一键。 |

重复事件不应当作为离线补偿失败。

### 错误处理

| 场景 | 处理 |
| --- | --- |
| 唯一键冲突 | 返回 Duplicate，不抛业务失败。 |
| 字段超长 | 截断允许截断的字段，并在 raw payload 记录。 |
| 必填字段缺失 | 不入库，返回 Invalid。 |
| 数据库连接失败 | 返回 RetryableFailure，事件消费重试或记录错误。 |
| SQL 超时 | 返回 RetryableFailure。 |

### 验收标准

- [ ] 正常入库：所有字段映射正确。
- [ ] 重复 EventId：第二次返回 Duplicate。
- [ ] 字段超长：被安全截断或失败原因明确。
- [ ] 无昵称：仍可入库。
- [ ] 数据库失败：返回可重试失败。
- [ ] 离线上传重复：Duplicate 视为处理成功。

## 任务 7.6：客户端布防与离线事件上传补偿

### 目标

实现设备恢复在线后的 ACS 离线事件上传补偿。阶段 7 只采用客户端布防方式：布防参数 `byDeployType = 0`，设备恢复后通过 SDK 报警回调补上传离线期间产生的 `COMM_ALARM_ACS` 事件，服务端通过 `byCurrentEvent == 2` 识别离线事件并写入 `dbo.attendance_gate_v2`。

阶段 7 不实现平台主动拉取历史事件，不调用 `NET_DVR_GET_ACS_EVENT`，不投递 `QueryHistoryEvents`，不读写 `dbo.face_event_checkpoint`。

### 布防方式

| 项目 | 要求 |
| --- | --- |
| SDK 接口 | `NET_DVR_SetupAlarmChan_V41`。 |
| 布防参数 | `NET_DVR_SETUPALARM_PARAM.byDeployType = 0`。 |
| 配置语义 | 如保留 `AlarmDeployType` 配置，阶段 7 只允许值 `0`；未配置或非法值回退为 `0`。 |
| 补偿来源 | 设备恢复在线后通过 `COMM_ALARM_ACS` 回调上传离线事件。 |
| 离线标记 | `NET_DVR_ACS_EVENT_INFO_EXTEND.byCurrentEvent == 2`。 |

### 离线事件识别

回调解析 `NET_DVR_ACS_ALARM_INFO` 时，应在扩展结构可用时读取 `NET_DVR_ACS_EVENT_INFO_EXTEND.byCurrentEvent`：

| `byCurrentEvent` | 处理 |
| --- | --- |
| `2` | 标记为设备上传的离线 ACS 事件，继续入队处理。 |
| `0` 或不可用 | 按普通实时 ACS 事件处理。 |
| 其他值 | 保留原值写入 raw payload，按实时事件处理或记录 debug，具体以 SDK 文档语义为准。 |

如果 `OfflineCompensationEnabled = false`，`byCurrentEvent == 2` 的事件应记录后忽略；普通实时事件仍可处理。

### 回调处理流程

| 步骤 | 动作 |
| --- | --- |
| 1 | 设备登录成功后执行客户端布防，确保 `byDeployType = 0`。 |
| 2 | SDK 回调收到 `COMM_ALARM_ACS`。 |
| 3 | 回调线程复制报警结构、抓拍图片和来源设备信息。 |
| 4 | 读取扩展字段 `byCurrentEvent`，判断是否为离线事件上传。 |
| 5 | 过滤非人脸/门禁认证事件。 |
| 6 | 构造 `RawAcsAlarmEvent`，将 `Source` 标记为 `Realtime` 或 `OfflineUpload`。 |
| 7 | 投递 `FaceEventIngestionService` 队列，回调线程立即返回。 |
| 8 | 后台消费者复用阶段 7.3-7.5 的解析、图片保存和入库逻辑。 |
| 9 | 唯一键冲突按重复事件处理，不重复写入。 |

### 不做的事

| 不做内容 | 原因 |
| --- | --- |
| 不调用 `NET_DVR_GET_ACS_EVENT` | 阶段 7 已限定为客户端布防离线事件上传。 |
| 不实现 `QueryHistoryEvents` | 不做平台主动历史查询。 |
| 不读写 `face_event_checkpoint` | 离线补偿进度由设备上传语义承担，服务端只做事件防重和入库。 |
| 不维护历史查询断点 | 没有主动拉取批次，也就没有断点前移语义。 |
| 不新增事件暂存表 | 保持阶段 7 表结构零变更。 |

### 验收标准

- [ ] 客户端布防参数：`NET_DVR_SETUPALARM_PARAM.byDeployType` 使用 `0`。
- [ ] 离线事件识别：`byCurrentEvent == 2` 被标记为 `OfflineUpload`。
- [ ] 离线事件入库：mock 离线 ACS 回调写入 `attendance_gate_v2`。
- [ ] 离线补偿关闭：`OfflineCompensationEnabled = false` 时忽略 `byCurrentEvent == 2`。
- [ ] 重复离线事件：第二次写入返回 Duplicate，不重复入库。
- [ ] 主动查询禁用：不调用 `NET_DVR_GET_ACS_EVENT`，不读写 `face_event_checkpoint`。

## 任务 7.7：队列缓冲、配置、测试与验收

### 目标

定义 ACS 实时事件与离线上传事件的队列缓冲策略、阶段 7 配置校验、自动化测试、真实设备联调和阶段通过标准。

### 队列缓冲策略

阶段 7 不做平台主动历史查询，也不存在历史补偿期间的专用实时缓冲。实时 ACS 事件和设备恢复后上传的离线 ACS 事件统一进入 `FaceEventIngestionService` 队列，后台消费者按接收顺序处理。

| 状态 | 行为 |
| --- | --- |
| 队列正常 | 实时事件和离线上传事件直接入队。 |
| 队列接近满 | 记录 `Warn`，继续投递。 |
| 队列已满 | 按配置丢弃或降级，并记录 `Error`。 |
| 服务停止 | 尽量处理已入队事件；未入队事件不持久化。 |

### 缓冲容量

| 项目 | 默认建议 |
| --- | --- |
| 全局队列容量 | 2000 条。 |
| 队列接近满阈值 | 80%。 |
| 超限策略 | 记录 `Error` 并丢弃新事件；如实现配置项，应保持默认行为明确。 |

### 配置校验

沿用 `FaceEventLogging`：

| 配置项 | 默认值 | 校验 |
| --- | --- | --- |
| `Enabled` | `true` | false 时不启动阶段 7 模块。 |
| `SnapshotRootDirectory` | `snapshots` | 不存在则创建，创建失败则模块不可用。 |
| `ExcludedDeviceIds` | `[]` | 只影响阶段 7 事件入库与离线上传补偿。 |
| `ExcludedDeviceIps` | `[]` | 只影响阶段 7 事件入库与离线上传补偿。 |
| `OfflineCompensationEnabled` | `true` | false 时忽略 `byCurrentEvent == 2` 的离线事件，普通实时事件仍处理。 |
| `AlarmDeployType` | `0` | 阶段 7 固定客户端布防；非法值回退为 `0` 或配置校验失败。 |
| `QueueCapacity` | `2000` | 小于最小值时使用默认值。 |

### 日志分类

| 事件 | 级别 | 说明 |
| --- | --- | --- |
| ACS 回调接收 | `Debug` | 正常回调。 |
| 事件入队 | `Debug` | 正常投递队列。 |
| 队列满丢弃 | `Error` | 需要关注容量。 |
| 设备无法识别 | `Warn` | 可能影响事件入库。 |
| 事件入库成功 | `Debug` | 正常。 |
| 重复事件跳过 | `Info` | 正常防重。 |
| 图片保存失败 | `Error` | 不影响入库但需关注磁盘。 |
| 离线事件上传 | `Info` | 记录设备、流水、`byCurrentEvent`。 |
| 离线事件被忽略 | `Warn` | 补偿关闭或设备被排除。 |
| 主动查询被禁用 | `Debug` | 不启动历史查询任务。 |

### 模块划分

| 模块 | 职责 |
| --- | --- |
| `FaceEventIngestionService` | 阶段 7 总入口，管理实时/离线上传事件队列和生命周期。 |
| `AcsAlarmEventRouter` | 从 SDK 原始回调识别 ACS 事件、反查设备、投递队列。 |
| `AcsEventParser` | 解析员工编号、事件时间、流水、方向、设备、认证结果和图片。 |
| `SnapshotStorage` | 保存抓拍图片，生成相对或绝对路径。 |
| `FaceEventRepository` | 写入 `attendance_gate_v2`、查询防重、补全昵称。 |
| `OfflineAcsEventPolicy` | 识别 `byCurrentEvent == 2` 离线事件，并按配置决定接收或忽略。 |

### 主流程

| 步骤 | 动作 |
| --- | --- |
| 1 | 阶段 4 完成设备登录和 ACS 客户端布防，`byDeployType` 固定使用 `0`。 |
| 2 | SDK 回调收到实时或设备恢复后上传的 `COMM_ALARM_ACS`。 |
| 3 | 回调线程复制报警结构、图片字节、来源设备信息。 |
| 4 | 识别 `byCurrentEvent`：普通实时事件直接入队，`byCurrentEvent == 2` 标记为离线上传事件后入队。 |
| 5 | 后台消费者解析为标准 `AcsFaceEvent`。 |
| 6 | 保存抓拍图片。 |
| 7 | 生成 `attendance_gate_v2` 记录并插入。 |
| 8 | 若业务 `id` 已存在，按重复事件处理，不重复入库。 |

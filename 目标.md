# 阶段 4：设备生命周期与设备管理 gRPC — 实施目标与任务清单

## 概述

阶段 4 的目标是把**阶段 2 的设备固定执行通道**与**阶段 3 的海康 SDK 网关**接入实际设备生命周期，并实现 `device.AccessControlService` 的 5 个设备管理 gRPC 方法。

阶段 4 **不实现权限/人员/人脸同步、不写离线补偿表、不做事件入库、不处理摄像头联动**。它只负责设备的完整生命周期管理：

- 从 `dbo.devices` 加载启用设备，注册到 `DeviceRuntimeRegistry`；
- 通过 `DeviceSdkDispatcher` 按设备固定通道串行执行登录、登出、状态检测、重连、布防、撤防；
- 通过既定 gRPC 契约对外提供 `GetDeviceStatus`、`AddDevice`、`DeleteDevice`、`DisconnectDevice`、`ReconnectDevice`。

## 硬性边界

| 边界 | 要求 |
| --- | --- |
| gRPC 完全兼容 | 服务名、方法名、方法类型、请求 JSON、响应 JSON、错误码和 `x-api-key` 约定保持不变。 |
| 数据库零变更 | 不修改任何现有表结构；阶段 4 只读写 `dbo.devices` 既有字段。 |
| 设备操作通道化 | 登录、登出、状态检测、布防、撤防、重连必须进入 `DeviceSdkDispatcher`。 |
| 不做权限/人员/人脸业务 | 权限、人员、人脸同步属于阶段 5，阶段 4 仅管理设备连接与生命周期。 |
| 不写离线补偿 | 设备生命周期失败只更新内存状态和日志；写补偿属于阶段 6。 |
| 不做事件入库 | 报警回调原始事件可投递，ACS 解析和入库属于阶段 7。 |
| 不做摄像头联动 | AIOP 联动门禁常闭属于阶段 9。 |
| 依赖阶段 3 网关 | 所有 SDK/ISAPI 调用必须通过 `IHikvisionGateway` 接口，不得直接调用 `HCNetSDK`。 |
| 依赖阶段 2 通道 | 所有设备操作必须通过 `DeviceSdkDispatcher` 投递为 `DeviceSdkTask`。 |

## 任务 4.2：设备加载与运行时注册

### 目标

从 `dbo.devices` 加载启用设备记录，转换为 `DeviceRuntimeState` 内存对象，并注册到 `DeviceRuntimeRegistry` 的 ID、IP、UserID、AlarmHandle 反查索引。**本任务只负责加载和注册，不登录设备、不布防、不调用真实 SDK。**

### 数据来源

| 字段 | 来源 | 用途 |
| --- | --- | --- |
| `device_id` | `dbo.devices.device_id` | 设备主键、任务路由键。 |
| `device_name` | `dbo.devices.device_name` | 状态展示和日志。 |
| `description` | `dbo.devices.description` | 状态展示。 |
| `ip_address` | `dbo.devices.ip_address` | SDK 登录地址和 IP 反查。 |
| `port` | `dbo.devices.port` | SDK 登录端口。 |
| `username` | `dbo.devices.username` | SDK 登录用户名。 |
| `password` | `dbo.devices.password` | SDK 登录密码。 |
| `status` | `dbo.devices.status` | `1` 表示启用。 |
| `last_used_time` | `dbo.devices.last_used_time` | 最近使用时间。 |

### 加载规则

- 启动时执行 `SELECT ... FROM dbo.devices WHERE status = 1 ORDER BY device_id`。
- 单条记录校验：`device_id > 0`、IP 非空、端口合法（1-65535）。
- 非法记录标记为 `InvalidConfig`，不投递登录。
- IP 冲突时保留第一条，冲突设备标记错误。
- 生成加载摘要日志：成功 N、跳过 M、冲突 K、非法 J。

### 验收标准

- [x] 启动时能读取 `status = 1` 的设备记录并注册内存运行时。
- [x] 无效设备不阻塞有效设备加载。
- [x] IP 冲突正确检测并记录警告。
- [x] 加载摘要日志包含成功/跳过/冲突/非法四种计数。

## 任务 4.3：启动登录与登出清理

### 目标

把阶段 3 网关的登录/登出方法接入设备固定执行通道。启动时为启用设备投递登录任务；删除、断开、重连和服务停止时投递登出清理任务。

### 登录任务

| 字段 | 说明 |
| --- | --- |
| `OperationName` | `DeviceLogin` |
| `Priority` | `High`（高于状态检测，低于强制断开/删除） |
| `TimeoutMs` | `DeviceConnection.LoginTimeoutMs`（默认 10 秒） |
| `Execute` | 调用 `IHikvisionGateway.LoginAsync` |

### 登录流程

| 步骤 | 动作 | 失败处理 |
| --- | --- | --- |
| 1 | 读取设备当前状态和连接参数 | 设备不存在返回 `NOT_FOUND` |
| 2 | 已在线且 UserID 有效 → 跳过重复登录 | 返回当前快照 |
| 3 | 设置状态为 `Connecting` | 写内存状态 |
| 4 | 调用 `LoginAsync` | 失败记录 SDK 错误，进入重连策略 |
| 5 | 写入 `SdkUserId`、设备序列号、索引 | 索引失败尝试登出 |
| 6 | 状态更新为 `Online` | 记录 `LastCheckedAt` |
| 7 | 更新 `last_used_time = SYSDATETIME()` | DB 失败不登出，记录 `DB_ERROR` |
| 8 | 按任务 4.5 投递布防任务 | 布防失败不影响登录成功 |

### 登出任务

| 场景 | 优先级 | 行为 |
| --- | --- | --- |
| 服务停止 | `Critical` | best-effort 撤防后登出 |
| 删除设备 | `Critical` | 撤防、登出、清理索引、删除数据库记录 |
| 手动断开 | `Critical` | 撤防、登出、状态置为 `Disconnected` |
| 强制重连 | `Critical` | 撤防、登出、清理旧 UserID 后重新登录 |

### 验收标准

- [x] 登录成功 → UserID 写入状态和反查索引，状态为 `Online`。
- [x] 登录失败 → 状态为 `Offline`，错误码保存，进入重连策略。
- [x] 已在线设备不重复调用 SDK 登录。
- [x] 登录后索引失败 → 尝试登出，避免 UserID 泄漏。
- [x] 登出成功 → UserID、AlarmHandle 和反查索引清空。
- [x] 登出失败 → 本地索引仍清理，错误进入日志。
- [x] 服务停止时所有在线设备完成撤防+登出（best-effort）。

## 任务 4.4：状态检测与重连策略

### 目标

实现周期性的设备在线状态检测和自动重连机制。检测失败后按指数退避策略调度重连，不阻塞其他设备的正常操作。

### 周期检测

| 项目 | 说明 |
| --- | --- |
| 检测间隔 | `DeviceConnection.HealthCheckIntervalMs`（默认 30 秒） |
| 检测方式 | 向设备投递 `DeviceHealthCheck` 任务（优先级 `Low`） |
| 在线判断 | 上次检测成功且在超时窗口内 |
| 离线判断 | 连续 N 次检测失败（默认 3 次） |

### 重连策略

| 项目 | 说明 |
| --- | --- |
| 退避算法 | 指数退避：1s → 2s → 4s → 8s → 16s → 32s（上限 60s） |
| 最大重试 | `DeviceConnection.MaxReconnectAttempts`（默认 10 次） |
| 手动重连 | `ReconnectDevice` 重置退避计数器，立即尝试 |
| 重连成功 | 退避计数器归零，状态恢复 `Online` |
| 重连耗尽 | 状态置为 `Failed`，记录日志，停止自动重连 |

### 验收标准

- [x] 周期性检测任务按配置间隔投递。
- [x] 连续 3 次检测失败判定为离线。
- [x] 自动重连按指数退避调度，不阻塞其他设备。
- [x] 手动重连 `ReconnectDevice` 立即重置退避并尝试连接。
- [x] 重连成功恢复在线，重连耗尽标记 `Failed`。

## 任务 4.5：布防编排与撤防策略

### 目标

实现登录后自动布防、断开前自动撤防的编排逻辑，管理 AlarmHandle 的生命周期和反查索引。

### 布防编排

| 项目 | 说明 |
| --- | --- |
| 触发时机 | 设备登录成功（`Online`）后立即投递 |
| 任务类型 | `DeviceArmAlarm`，优先级 `Normal` |
| 布防模式 | 按设备能力选择：ACS 门禁事件、人脸对比事件、AIOP 视频事件 |
| 回调注册 | 布防成功后注册 AlarmHandle → device_id 反查索引 |

### 撤防策略

| 触发时机 | 行为 |
| --- | --- |
| 设备断开前 | 先撤防再登出 |
| 设备删除前 | 先撤防再登出再删库 |
| 服务停止前 | best-effort 撤防所有在线设备 |
| 重连前 | 撤防旧连接再登录新连接 |

### AlarmHandle 管理

- 布防成功 → 注册 `AlarmHandle → deviceId` 反查索引。
- 撤防成功 → 清理反查索引。
- 撤防失败 → 标记 `AlarmHandle` 为已失效，本地仍清理索引。
- SDK 回调 → 通过 AlarmHandle 反查设备 ID，投递事件到后台队列。

### 验收标准

- [x] 登录成功后自动投递布防任务。
- [x] 布防成功建立 AlarmHandle 反查索引。
- [x] 断开/删除/停止前先执行撤防再登出。
- [x] 撤防失败本地仍清理索引，防止误用失效句柄。
- [x] 重连场景先撤防旧连接再布防新连接。

## 任务 4.6：`GetDeviceStatus`

### 目标

实现 `device.AccessControlService/GetDeviceStatus` gRPC handler，支持按设备 ID、ID 列表、IP 或全部查询，支持可选立即刷新和停用设备包含。

### gRPC 契约

| 项目 | 内容 |
| --- | --- |
| 完整方法名 | `/device.AccessControlService/GetDeviceStatus` |
| 方法类型 | Unary，`string -> string` |
| 鉴权 | 受 `x-api-key` 保护 |

### 请求字段

| 字段 | 别名 | 必填 | 说明 |
| --- | --- | --- | --- |
| `deviceId` | `device_id` | 否 | 查询单台设备 |
| `deviceIds` | `device_ids` | 否 | 查询多台设备 |
| `ipAddress` | `ip_address` | 否 | 按 IP 查询 |
| `includeDisabled` | — | 否 | 是否包含停用设备，默认 true |
| `refresh` | — | 否 | 是否立即刷新状态，默认 false |

### 响应字段

`devices[]` → `deviceId`、`deviceName`、`ipAddress`、`port`、`enabled`、`isConnected`、`status`、`statusMessage`、`lastChecked`、`lastUsed`、`lastErrorCode`、`lastErrorMessage`。

### 验收标准

- [x] 空请求 `{}` 返回全部设备状态。
- [x] 按 `deviceId`、`deviceIds`、`ipAddress` 精确筛选。
- [x] `refresh=true` 触发即时状态检测（不等待周期）。
- [x] `includeDisabled=false` 过滤停用设备。
- [x] 不存在的设备返回 `NOT_FOUND` 错误码，不影响其他设备。

## 任务 4.7：`AddDevice`

### 目标

实现 `device.AccessControlService/AddDevice` gRPC handler，新增设备到数据库和内存运行时，支持可选立即连接。

### gRPC 契约

| 项目 | 内容 |
| --- | --- |
| 完整方法名 | `/device.AccessControlService/AddDevice` |
| 方法类型 | Unary，`string -> string` |
| 鉴权 | 受 `x-api-key` 保护 |

### 请求字段

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `deviceId` | 是 | 大于 0 的整数，数据库主键非自增 |
| `deviceName` | 是 | 设备名称 |
| `ipAddress` | 是 | 设备 IP |
| `port` | 否 | 端口，默认 `8000` |
| `username` | 否 | 登录用户名，默认 `admin` |
| `password` | 是 | 登录密码 |
| `description` | 否 | 设备描述 |
| `enabled` | 否 | 是否启用，默认 true |
| `connectNow` | 否 | 新增后是否立即连接，默认 false |

### 执行流程

| 步骤 | 动作 | 失败处理 |
| --- | --- | --- |
| 1 | 校验必填字段 | 返回 `INVALID_ARGUMENT` |
| 2 | 检查 `deviceId` 是否已存在 | 已存在返回 `INVALID_ARGUMENT` |

| 6 | `connectNow=true` → 投递登录任务 | 登录失败不影响新增成功 |

### 验收标准

- [x] 必填字段缺失返回 `INVALID_ARGUMENT`。
- [x] 重复 `deviceId` 返回错误。
- [x] 新增成功返回设备信息和连接状态。
- [x] `connectNow=true` 异步触发登录，不阻塞响应。

3. **任务 4.3**（启动登录与登出）— 接入阶段 3 网关，实现登录/登出编排。
4. **任务 4.4**（状态检测与重连）— 依赖任务 4.3，增加周期检测和自动重连。
5. **任务 4.5**（布防与撤防）— 依赖任务 4.3，实现登录后布防、断开前撤防。
6. **任务 4.6**（`GetDeviceStatus`）— 依赖任务 4.2/4.4，实现第一个 gRPC handler。
7. **任务 4.7**（`AddDevice`）— 依赖任务 4.2/4.3，实现设备新增。
8. **任务 4.8**（`DeleteDevice`、`DisconnectDevice`、`ReconnectDevice`）— 依赖任务 4.3/4.5。
9. **任务 4.9**（测试与验收）— 贯穿全阶段，最终汇总验证。

## 阶段 4 完成标准

| 标准 | 说明 |
| --- | --- |
| 设备可加载 | 启动时能读取 `status = 1` 的 `dbo.devices` 记录并注册内存运行时。 |
| 登录可编排 | 登录成功更新运行时 UserID 和状态；登录失败进入离线/重连策略。 |
| 清理可控 | 删除、断开、服务停止时按撤防 → 登出 → 索引清理顺序执行。 |
| gRPC 可用 | 5 个设备管理方法均按 `docs/gRPC接口清单.md` 的契约返回。 |
| 数据库兼容 | 不新增或修改表结构，只使用 `dbo.devices` 既有字段。 |
| 通道化执行 | 所有设备操作通过 `DeviceSdkDispatcher` 投递，同一设备串行执行。 |
| 测试可证明 | mock gateway 可覆盖成功、失败、超时、重连、布防失败等关键路径。 |

## 后续阶段衔接

阶段 4 完成后，设备已具备完整的连接生命周期管理能力。后续阶段的使用方式为：

```csharp
// 阶段 5 权限同步 — 使用已登录设备的 UserId
var state = _registry.GetByDeviceId(deviceId);
if (state.IsOnline)
{
    await _gateway.SetPermissionAsync(new SetPermissionRequest
    {
        UserId = state.SdkUserId.Value,
        Permissions = permissions
    });
}
else
{
    // 离线设备写入补偿队列（阶段 6）
    _compensationWriter.Enqueue(deviceId, operation);
}

// 阶段 7 事件回调 — AlarmHandle 反查设备 ID
// 布防回调触发时通过 AlarmHandle → deviceId 索引定位设备
_gateway.OnAlarmEvent += (sender, data) =>
{
    var deviceId = _registry.GetDeviceIdByAlarmHandle(data.AlarmHandle);
    _eventQueue.Enqueue(new DeviceEvent { DeviceId = deviceId, Data = data });
};
```

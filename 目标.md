# 阶段 5：权限、人员、人脸同步 — 实施目标与任务清单

## 概述

阶段 5 的目标是实现 `permission.PermissionSyncService` 的 7 个 gRPC 方法，完成权限下发、人员同步、人脸管理、人脸采集的完整链路。

阶段 5 **不修改数据库表结构、不实现补偿扫描与重试、不做事件入库、不处理摄像头联动**。它只负责：

- 解析 gRPC 请求，识别容器格式和字段别名；
- 选择目标在线设备，通过 `DeviceSdkDispatcher` 调用阶段 3 网关的 SDK/ISAPI 能力；
- 离线设备产出补偿意图（阶段 6 负责扫描和重试）；
- 返回员工维度、设备维度的结果明细。


## 硬性边界

| 边界 | 要求 |
| --- | --- |
| gRPC 完全兼容 | 服务名、方法名、方法类型、请求 JSON、响应 JSON、错误码和 `x-api-key` 约定保持不变。 |
| 数据库零结构变更 | 不修改任何现有表结构；阶段 5 可更新 `system_users` 既有门禁同步字段，不新增字段。 |
| 批量上限 | 权限、人员、人脸、删除类接口最大 500 条，超出返回 `BATCH_TOO_LARGE`。 |
| 同步操作通道化 | 权限下发、人员同步、人脸操作、删除等设备写入必须进入 `DeviceSdkDispatcher`。 |
| 补偿只产出意图 | 阶段 5 识别离线或可重试失败并调用补偿记录接口；补偿扫描、重试、终态和清理由阶段 6 实现。 |
| 不做事件入库 | 报警回调原始事件可投递，ACS 解析和入库属于阶段 7。 |
| 不做摄像头联动 | AIOP 联动门禁常闭属于阶段 9。 |
| 依赖阶段 3 网关 | 人员 CRUD、人脸上传/删除/查询、权限下发均通过 `IHikvisionGateway` 接口。 |
| 依赖阶段 4 设备管理 | 通过 `DeviceRuntimeRegistry` 获取在线设备快照，决定直接下发还是补偿排队。 |

## 任务 5.2：`SyncPermissions`

### 目标

实现 `/permission.PermissionSyncService/SyncPermissions`，按员工编号把权限编号同步到在线门禁设备。支持数组、`items`、`records` 和单个对象请求结构。

### gRPC 契约

| 项目 | 内容 |
| --- | --- |
| 完整方法名 | `/permission.PermissionSyncService/SyncPermissions` |
| 方法类型 | Unary，`string -> string` |

### 请求字段

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `employee_id` | 是 | 员工编号/工号 |
| `permission_code` | 是 | 权限编号，必须可解析为整数 |

### 解析规则

| 输入结构 | 处理 |
| --- | --- |
| JSON 数组 | 每个元素解析为一条权限同步记录 |
| `{"items": [...]}` | 使用 `items` |
| `{"records": [...]}` | 使用 `records` |
| 单个对象 | 解析为一条记录 |

### 执行流程

| 步骤 | 动作 | 失败处理 |
| --- | --- | --- |
| 1 | 校验批量 ≤ 500 | 返回 `BATCH_TOO_LARGE` |
| 2 | 校验必填字段 | 返回 `INVALID_ARGUMENT` |
| 3 | 合并同一员工的重复记录，保留最后一次权限值 | — |
| 4 | 获取启用且在线的设备快照 | — |
| 5 | 对每台在线设备投递 `SyncPermission` 任务 | SDK 失败记录 `DEVICE_ERROR` |
| 6 | 离线设备生成补偿意图 | 补偿接口失败记录警告 |
| 7 | 更新 `system_users` 的 `access_permission`、`permission_updated_at` | DB 失败不影响设备结果 |
| 8 | 汇总 `total`、`updated`、`skipped`、`failed`、`queued`、`queuedDetails` | — |

### 验收标准

- [x] 支持数组、`items`、`records`、单对象四种输入结构。
- [x] 同员工重复记录合并为最后一次权限值。
- [x] 在线设备直接下发、离线设备补偿排队。
- [x] 响应包含 `total`、`updated`、`skipped`、`failed`、`queued`、`queuedDetails`。
- [x] 批量超 500 返回 `BATCH_TOO_LARGE`。

## 任务 5.3：`SyncPersons`

### 目标

实现 `/permission.PermissionSyncService/SyncPersons`，向在线设备下发人员基础信息，并在包含人脸图片时下发人脸。**人员必须先于人脸下发，人员失败则跳过该设备的人脸。**

### gRPC 契约

| 项目 | 内容 |
| --- | --- |
| 完整方法名 | `/permission.PermissionSyncService/SyncPersons` |
| 方法类型 | Unary，`string -> string` |

### 请求字段

| 字段 | 别名 | 必填 | 说明 |
| --- | --- | --- | --- |
| `employee_id` | `employeeId`、`employee_no`、`employeeNo` | 是 | 员工编号 |
| `name` | `full_name`、`fullName` | 否 | 姓名 |
| `gender` | `sex` | 否 | 性别，建议 `male`/`female`/`unknown` |
| `enabled` | `active`、`is_active` | 否 | 是否启用，默认 true |
| `valid_from` | `validFrom` | 否 | 有效期开始 |
| `valid_to` | `validTo` | 否 | 有效期结束 |
| `face_image_base64` | `faceImageBase64`、`face_base64`、`faceBase64`、`face_image` | 否 | 人脸图片 Base64，支持 data URI |
| `face_image_format` | `faceImageFormat` | 否 | 图片格式，仅用于日志和结果说明 |

### 校验规则

| 项目 | 规则 |
| --- | --- |
| 批量数量 | 最大 500 |
| 员工编号 | 非空 |
| 有效期 | 可空；如都存在，开始不得晚于结束 |
| 人脸 Base64 | 支持 data URI 前缀；解码失败返回 `INVALID_ARGUMENT` |
| 人脸大小 | 解码后超过 200KB 返回 `FACE_TOO_LARGE` |

### 执行流程

| 步骤 | 动作 | 失败处理 |
| --- | --- | --- |
| 1 | 解析请求容器 `people`/`items`/`records`/`data` 或单对象 | 返回 `INVALID_ARGUMENT` |
| 2 | 标准化人员字段和人脸图片 | — |
| 3 | 获取目标在线设备快照 | — |
| 4 | 对每台设备先投递人员下发任务 | SDK 失败跳过该设备后续步骤 |
| 5 | 人员成功后，若有人脸图片，投递人脸下发任务 | 人脸失败不影响人员成功计数 |
| 6 | 离线或可重试失败产生补偿意图 | 记录排队明细 |
| 7 | 汇总 `total`、`succeeded`、`failed`、`queued`、`facesUploaded`、`queuedDetails` | — |

### 设备任务

| 字段 | 说明 |
| --- | --- |
| 人员下发 | `SyncPerson`，优先级 `Normal` |
| 人脸下发 | `UploadFace`，优先级 `Normal`（人员成功后才投递） |

### 设备端实现边界

| 能力 | 说明 |
| --- | --- |
| 人员下发 | 可用 ISAPI 或远程配置 |
| 人脸下发 | 可用 `NET_DVR_FACE_PARAM_*` 远程配置或 ISAPI |
| 设备不支持人脸 | 返回设备维度失败，不阻塞人员下发 |
| 日志控制 | 人脸 Base64 是否完整记录由配置控制 |

### 验收标准

- [x] 支持 `people`/`items`/`records`/`data` 及单对象容器。
- [x] 人员先于人脸下发，人员失败跳过该设备人脸。
- [x] 无人脸图片时仅下发人员信息。
- [x] 人脸超 200KB 返回 `FACE_TOO_LARGE`。
- [x] 响应包含 `facesUploaded`、`targetDevices` 和设备维度明细。

## 通用处理流程

所有 7 个 gRPC 方法共享以下通用处理逻辑，减少重复代码：

| 步骤 | 动作 | 说明 |
| --- | --- | --- |
| 1 | 读取 requestId | 从 `x-request-id` 头或生成新 ID |
| 2 | 解析 JSON 字符串 | 反序列化为 `JToken` 或 `JsonElement` |
| 3 | 识别请求容器 | 检测数组 / `items` / `records` / `people` / `data` / 单对象 |
| 4 | 字段别名映射 | `employee_id` ↔ `employeeId` ↔ `employee_no` ↔ `employeeNo` 等 |
| 5 | 校验批量上限 | 超过 500 返回 `BATCH_TOO_LARGE` |
| 6 | 校验必填字段 | 逐个校验，失败返回 `INVALID_ARGUMENT` |
| 7 | 标准化为业务命令 | 统一为 `List<SyncCommand>` 或等价结构 |
| 8 | 获取目标设备快照 | 从 `DeviceRuntimeRegistry` 获取启用设备列表 |
| 9 | 设备分类 | 在线 → 直接下发；离线 → 补偿排队；停用 → 跳过 |
| 10 | 汇总结果 | 聚合 `total`、`succeeded`、`failed`、`skipped`、`queued` |
| 11 | 更新 `system_users` | 按接口语义更新既有门禁同步字段 |
| 12 | 返回统一 JSON 响应 | 包含 `requestId`、`success`、`code`、`message` 等标准字段 |

## 目标设备选择规则

| 场景 | 规则 | 计入 |
| --- | --- | --- |
| 在线设备 | 立即投递设备任务到 `DeviceSdkDispatcher` | `succeeded` 或 `failed` |
| 离线设备 | 不调用 SDK，产出补偿意图写入 `device_operation_retry_states` | `queued` |
| 停用设备 | 不参与同步，不写入补偿 | `skipped` |
| 配置非法设备 | 不参与同步 | `skipped` 或 `failed` |

## system_users 更新规则

| 接口 | 更新字段 | 时机 |
| --- | --- | --- |
| `SyncPermissions` | `access_permission`、`permission_updated_at`、`last_synced_at` | 全部在线设备成功后 |
| `SyncPersons` | `last_synced_at` | 人员下发成功后 |
| `DeletePersons` | 相关同步字段置空或标记 | 删除成功后 |

约束：不修改 `system_users` 表结构，不新增字段，不改变账号体系语义。

## 补偿意图规范

阶段 5 只负责**产出补偿意图**，不负责扫描、重试、终态和清理（由阶段 6 实现）。

| 字段 | 说明 |
| --- | --- |
| `device_id` | 目标设备 ID |
| `operation` | `SyncPermission` / `SyncPerson` / `UploadFace` / `DeleteFace` / `DeletePerson` |
| `employee_id` | 员工编号 |
| `payload` | 操作所需的完整 JSON 参数 |
| `created_at` | 补偿记录创建时间 |
| `status` | 初始 `pending` |

补偿意图写入通过 `IDeviceOperationRetryWriter` 接口完成，阶段 5 不直接操作 `device_operation_retry_states` 表。

## 通用响应结构

所有 Unary 方法必须包含：

```json
{
    "requestId": "请求ID",
    "success": true,
    "code": "OK",
    "message": "处理结果说明",
    "errors": [],
    "errorDetails": []
}
```

不同接口追加的业务字段：

| 接口 | 追加业务字段 |
| --- | --- |
| `SyncPermissions` | `total`、`updated`、`skipped`、`failed`、`queued`、`queuedDetails` |
| `SyncPersons` | `total`、`succeeded`、`failed`、`queued`、`facesUploaded`、`targetDevices`、`queuedDetails` |
| `DeleteFaces` | `total`、`succeeded`、`failed`、`queued`、`targetDevices`、`queuedDetails`、`items` |
| `DeletePersons` | 同上 |
| `GetFaces` | `total`、`succeeded`、`failed`、`targetDevices`、`items` |
| `GetEnrollmentStatus` | `taskId`、`employeeId`、`action`、`status`、`message`、`errorCode` |

## 实施顺序

1. **任务 5.1**（边界总览）— 先行固化通用处理流程、设备选择规则和补偿意图规范。
2. **任务 5.2**（`SyncPermissions`）— 权限同步是最简单的写入链路，先实现以建立通用框架。
3. **任务 5.3**（`SyncPersons`）— 人员+人脸复合链路，验证人员先于人脸的编排逻辑。
4. **任务 5.4**（`DeleteFaces`、`DeletePersons`）— 删除链路，可与任务 5.3 并行。
5. **任务 5.5**（`GetFaces`）— 只读查询链路，可与任务 5.2-5.4 并行。
6. **任务 5.6**（`CaptureFaceStream`、`GetEnrollmentStatus`）— 流式接口，需要特殊处理。
7. **任务 5.7**（测试与验收）— 贯穿全阶段，最终汇总验证。
